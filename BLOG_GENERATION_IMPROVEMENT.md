# 공공데이터 블로그 생성 로직 개선안

## 문서 목적

이 문서는 현재 공공데이터 기반 블로그 생성 흐름을 운영 가능한 수준으로 개선하기 위한 제안만 정리한 문서입니다.
핵심 방향은 다음 한 문장으로 요약됩니다.

> 사실 추출은 최대한 결정적으로 처리하고, 문장화만 AI에 맡긴다.

---

## 핵심 제안

### 1. 지역 축을 먼저 고정

- 서비스의 기준 지역을 명시적으로 하나로 고정합니다.
- 후보:
  - 서울
  - 성남
  - 전국 정책형 정보
- 이후 수집 필터, 메인 데이터, 블로그 주제, UI 문구를 모두 같은 축으로 맞춥니다.

### 2. AI를 데이터 정규화 단계에서 제거 또는 축소

- 현재 구조는 `공공데이터 -> AI 가공 JSON -> AI 블로그 생성` 형태입니다.
- 이 구조는 오차가 두 번 누적될 가능성이 큽니다.
- 개선 방향:
  - 가능한 필드는 규칙 기반으로 직접 정규화
  - AI는 최종 글쓰기 단계에서만 사용
  - AI가 추정한 값은 데이터 원본에 저장하지 않음

### 3. 운영용 데이터 스키마로 확장

- 화면 표시용 필드만으로는 운영 자동화가 약합니다.
- 최소 추가 권장 필드:
  - `source_id`
  - `source_url`
  - `provider`
  - `region`
  - `published_at`
  - `updated_at`
  - `apply_start`
  - `apply_end`
  - `is_always_open`
  - `content_hash`
  - `status`

### 4. "최신 항목" 판정을 배열 순서가 아니라 기준 필드로 수행

- 블로그 생성 대상은 배열의 마지막 항목이 아니라 아래 기준으로 뽑아야 합니다.
  - `updated_at` 우선
  - 없으면 `published_at`
  - 동일 항목 여부는 `source_id` 또는 `content_hash`로 판정
- 이렇게 해야 실제 최신성과 중복 방지가 맞춰집니다.

### 5. 블로그 생성 프롬프트를 자유 생성형에서 근거 기반 템플릿형으로 변경

- AI에게 감성 확장과 추정 서술을 맡기지 않습니다.
- 아래처럼 고정 섹션을 채우는 방식이 더 안전합니다.
  - 제도/행사 한눈에 보기
  - 대상
  - 기간
  - 신청 또는 참여 방법
  - 주의사항
  - 공식 링크
- 원문에 없는 내용은 생성하지 않고 `공식 페이지 확인 필요`로 남깁니다.

### 6. 출처와 기준일을 본문 구조에 강제

- 모든 글 하단에 최소한 아래 정보를 강제로 넣습니다.
  - 공식 출처
  - 원문 URL
  - 정보 기준일
- 공공정보형 블로그는 문장력보다 신뢰도와 검증 가능성이 중요합니다.

### 7. 발행 단계를 `draft -> review -> publish`로 분리

- 자동 생성 결과를 바로 배포하지 않습니다.
- 권장 흐름:
  1. 데이터 수집
  2. 초안 생성
  3. 검수 또는 품질 게이트 통과
  4. 발행
- 최소 수준으로는 GitHub Actions가 바로 `main`에 반영하지 않고 초안 PR을 만드는 구조가 적절합니다.

### 8. 만료 정보 관리 로직 추가

- 행사와 지원금은 시간이 지나면 정보 가치가 급감합니다.
- 각 항목과 글에 상태를 둡니다.
  - `active`
  - `upcoming`
  - `closing-soon`
  - `expired`
  - `unknown`
- 만료 콘텐츠는 목록에서 제외하거나 경고 배지를 붙입니다.

### 9. 중복 판정을 제목 포함 여부가 아니라 원본 식별자 중심으로 변경

- 제목이나 본문 문자열 검색 기반 중복 판정은 약합니다.
- 우선순위:
  1. `source_id`
  2. `source_url`
  3. `title + provider + apply_start + apply_end` 조합 해시
- 블로그 글도 동일 기준으로 재생성 여부를 판정합니다.

### 10. 워크플로우에 품질 게이트 추가

- 자동 생성 후 바로 커밋하지 않고 아래 검사를 먼저 통과시킵니다.
  - frontmatter 필수값 존재 여부
  - 날짜 형식 검증
  - URL 형식 검증
  - 공식 출처 존재 여부
  - 금지 표현 검사
  - 글 길이 최소 기준
- 하나라도 실패하면 발행하지 않고 검토 대상으로 보냅니다.

### 11. 로그를 운영 리포트 형태로 정리

- 콘솔 로그를 많이 찍는 것보다 실행 요약이 중요합니다.
- 매 실행마다 아래 항목을 남기는 구조를 권장합니다.
  - 수집 건수
  - 신규 건수
  - 중복 제외 건수
  - 초안 생성 건수
  - 발행 건수
  - 실패 건수
  - 실패 원인

### 12. 수익화는 마지막 단계로 미룸

- AdSense나 제휴 배너는 우선순위가 아닙니다.
- 먼저 확보해야 할 것은 아래 3가지입니다.
  - 데이터 일관성
  - 출처 명시
  - 만료 정보 관리
- 이 기준이 정리된 뒤에 광고를 붙이는 편이 맞습니다.

---

## 권장 아키텍처

```text
공공데이터 API
  -> 규칙 기반 정규화
  -> 중복 판정 / 상태 판정
  -> structured data 저장
  -> 템플릿 기반 AI 초안 생성
  -> 품질 게이트
  -> draft 저장 또는 PR 생성
  -> 검토 후 publish
```

---

## 우선순위 제안

### 1순위

- 지역 축 고정
- 데이터 스키마 개편
- AI 개입 단계 축소

### 2순위

- 최신/중복 판정 개선
- 출처와 기준일 강제
- 품질 게이트 추가

### 3순위

- 만료 관리
- draft/review/publish 분리
- 운영 리포트 정리

### 후순위

- 광고 및 제휴 수익화

---

## 실행 원칙

- 원문에 없는 정보는 쓰지 않습니다.
- 날짜가 불명확하면 추정하지 않습니다.
- 공식 링크가 없으면 발행 우선순위를 낮춥니다.
- 한 번 자동화한 뒤에는 운영 기준이 코드보다 더 중요합니다.

---

## 최종 제안

가장 먼저 결정해야 할 것은 기능이 아니라 제품 정체성입니다.

정체성이 고정되면 아래 3개를 먼저 처리하는 것이 가장 효과적입니다.

1. 지역 축 고정
2. 데이터 스키마 개편
3. AI를 최종 문장화 단계로 한정

이 3개가 정리되면, 이후 자동화와 블로그 품질 개선은 훨씬 단순해집니다.
