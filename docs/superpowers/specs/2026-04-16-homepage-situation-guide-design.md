# 홈페이지 상황별 나들이 가이드 기획

작성일: 2026-04-16
작성 흐름: `/office-hours`
브랜치: `main`
상태: Draft

## 목표

이 프로젝트의 1차 목표는 작은 수익형 서울 콘텐츠 사이트를 운영하면서, 검색 유입, AdSense 승인, 자동 콘텐츠 생성, 데이터 파이프라인, 제품 기획 역량을 함께 키우는 것이다.

현재 사이트는 행사, 혜택, 블로그, 검색을 모두 갖고 있지만 첫인상이 정보 포털에 가깝다. 앞으로는 사용자가 "서울에서 이번 주말 어디 가지?"라는 선택 피로를 줄일 수 있도록, 홈페이지의 중심을 **상황별 서울 나들이 선택 도구**로 바꾼다.

## 문제 정의

사용자는 서울 갈 곳 정보가 없어서 힘든 것이 아니다. 네이버, 구글, 인스타그램, 블로그, 지도 앱에 정보는 많다. 진짜 문제는 정보가 흩어져 있고, 지금 자신의 상황에 맞는 선택지로 정리되어 있지 않다는 점이다.

이 사이트가 풀어야 할 문제는 "서울 정보 부족"이 아니라 **검색 피로와 선택 피로**다.

## 1차 타깃 사용자

서울에서 주말 나들이를 찾는 사람을 1차 사용자로 둔다.

예시 상황:

- 아이와 갈 곳을 찾는 부모
- 비 오는 날 실내 장소를 찾는 사람
- 돈을 많이 쓰지 않고 반나절 보낼 곳을 찾는 사람
- 행사나 혜택보다 "오늘 어디를 가면 좋을지"를 먼저 알고 싶은 사람

행사와 혜택 데이터는 버리지 않는다. 다만 홈의 정체성을 잡는 주인공은 행사/혜택 목록이 아니라 상황별 코스 추천이다.

## 핵심 가치

검색 여러 번 하지 않아도, 서울 나들이 코스를 상황별로 정리해준다.

이 가치는 다음 기준을 만족해야 한다.

- 사용자의 실제 고민에서 출발한다.
- 장소 나열보다 선택 기준을 제공한다.
- 공식 확인 링크를 통해 신뢰도를 보강한다.
- 글 수가 적어도 페이지 자체가 얇은 링크 모음처럼 보이지 않아야 한다.

## 홈페이지 정보 구조

1차 홈 구조는 다음 순서를 따른다.

1. Hero
   - 메인 문구: `검색 여러 번 하지 않아도, 서울 나들이 코스를 상황별로 정리해드립니다.`
   - 보조 문구: `아이와, 비 오는 날, 무료·저비용처럼 실제 고민에서 출발해 공식 확인 링크와 함께 서울 코스를 고를 수 있게 돕습니다.`
2. 상황별 선택 카드
   - `아이와`
   - `비 오는 날`
   - `무료·저비용`
3. 통합 검색
   - 직접 찾고 싶은 사용자를 위한 보조 수단
4. 최신 추천 글
   - 상황별 코스 글 중심
5. 서울 행사·축제
   - 보조 정보
6. 서울 혜택
   - 보조 정보
7. 출처와 업데이트 안내

## 1차 상황별 페이지

처음부터 6개 상황을 모두 만들지 않는다. 페이지가 얇아지는 것을 피하기 위해 1차 버전은 검색 의도가 강하고 이미 콘텐츠가 있는 3개만 만든다.

### `/situations/kids`

목표: 아이와 서울에서 갈 곳을 찾는 부모가 빠르게 판단할 수 있게 한다.

구성:

- 아이와 갈 때 먼저 봐야 할 기준
- 실내/야외 선택 팁
- 연령대별 주의점
- 대표 글: 아이와 함께 가기 좋은 서울 주말 코스
- 방문 전 체크리스트
- 공식 확인 안내

### `/situations/rainy-day`

목표: 비 오는 날에도 일정이 무너지지 않도록 실내 중심 선택 기준을 제공한다.

구성:

- 비 오는 날 서울 나들이에서 피해야 할 실수
- 실내 장소 선택 기준
- 이동 동선 줄이는 팁
- 대표 글: 서울 비 오는 날 실내 나들이 추천
- 방문 전 체크리스트
- 공식 확인 안내

### `/situations/free`

목표: 비용 부담 없이 서울에서 반나절 보내고 싶은 사용자가 적절한 장소를 고르게 한다.

구성:

- 무료 명소를 고를 때 놓치기 쉬운 점
- 무료지만 만족도 높은 장소 기준
- 반나절 동선 추천
- 대표 글: 서울 무료 나들이 코스
- 방문 전 체크리스트
- 공식 확인 안내

## 페이지 구성 원칙

각 상황별 페이지는 "추천 글 목록"만 보여주면 안 된다. 글이 1개뿐이어도 페이지 자체가 선택을 돕는 짧은 가이드여야 한다.

기본 구성:

1. 이 상황에서 사람들이 겪는 고민
2. 이 페이지가 어떤 선택을 도와주는지
3. 대표 추천 글 1개 이상
4. 방문 전 체크리스트
5. 공식 확인 안내
6. 관련 태그 또는 다음 상황 연결

## 접근안 비교

### 접근 A: 최소 구현

홈에 상황 카드 3개를 추가하고, 각 카드는 기존 블로그 태그 검색 또는 블로그 목록 필터로 이동한다.

장점:

- 가장 빠르게 만들 수 있다.
- 코드 변경이 적다.

단점:

- 사이트 정체성이 강해지지 않는다.
- SEO용 상황별 랜딩 페이지가 생기지 않는다.
- AdSense 관점에서 독립적인 콘텐츠 가치가 약하다.

### 접근 B: 권장안, 상황별 전용 페이지 3개

홈 상단을 상황 선택형으로 바꾸고, 3개 상황별 페이지를 만든다. 각 페이지는 자체 설명, 대표 글, 체크리스트, 공식 확인 안내를 가진다.

장점:

- 사용자의 실제 선택 흐름과 맞다.
- 검색 유입용 랜딩 페이지가 생긴다.
- 기존 블로그 글을 재사용하면서도 페이지 자체 가치가 생긴다.
- 나중에 행사, 혜택, 날씨, 지도와 연결하기 좋다.

단점:

- 홈 구조와 라우트 추가가 필요하다.
- 상황별 페이지 문구 품질을 신경 써야 한다.

### 접근 C: 상황별 선택 도구

상황 페이지에서 실내/야외, 무료/유료, 반나절/하루 같은 선택 버튼을 제공하고 추천 글이나 코스를 필터링한다.

장점:

- 제품처럼 느껴진다.
- 사용자가 선택받는 경험이 강하다.

단점:

- 현재 콘텐츠 수가 부족하다.
- 로직이 먼저 생기면 내용이 빈약해질 수 있다.
- 지금 단계에서는 과하다.

## 권장 접근

접근 B를 선택한다.

지금 필요한 것은 정교한 추천 엔진이 아니라, 사이트의 정체성을 선명하게 만드는 것이다. 상황별 전용 페이지 3개는 작은 범위로도 사이트의 방향을 바꿀 수 있고, 기존 블로그 콘텐츠와 자동 생성 파이프라인을 그대로 활용할 수 있다.

## SEO와 AdSense 관점

AdSense 승인을 보장하는 구조는 없다. 하지만 얇은 자동 생성 글을 나열하는 것보다, 상황별 페이지가 독립적인 선택 기준과 체크리스트를 가지는 편이 훨씬 안전하다.

SEO 관점에서도 `서울 아이와 갈만한 곳`, `서울 비 오는 날 실내`, `서울 무료 나들이` 같은 검색 의도를 직접 받을 수 있는 페이지가 생긴다.

중요한 기준:

- 링크 모음 페이지처럼 보이지 않게 한다.
- 공식 출처 안내를 유지한다.
- 각 페이지에 자체 설명과 방문 전 체크리스트를 넣는다.
- 블로그 글은 "더 읽기"가 아니라 실제 의사결정의 근거로 연결한다.

## 비목표

이번 1차 기획에서는 다음을 하지 않는다.

- 날씨 기반 자동 추천
- 지도 기반 거리 계산
- 개인화 추천
- 로그인이나 저장 기능
- 전체 카테고리 6개 동시 오픈
- DB 기반 동적 관리

이 기능들은 콘텐츠가 충분히 쌓인 뒤 검토한다.

## 성공 기준

1차 구현의 성공 기준은 다음과 같다.

- 홈 첫 화면에서 사이트의 목적이 "서울 나들이 상황별 선택"으로 보인다.
- `/situations/kids`, `/situations/rainy-day`, `/situations/free`가 각각 독립적인 정보 가치를 가진다.
- 각 페이지가 최소 1개 이상의 관련 블로그 글을 연결한다.
- 각 페이지에 방문 전 체크리스트가 있다.
- 행사/혜택은 홈에서 사라지지 않지만 보조 섹션으로 내려간다.
- 검색 인덱스나 기존 블로그/행사/혜택 흐름을 깨지 않는다.

## The Assignment

다음 작업 전에 확인할 것은 하나다.

사용자가 홈에 들어왔을 때 "이 사이트는 서울 행사/혜택 포털"로 보이길 원하는가, 아니면 "서울 나들이를 상황별로 고르게 해주는 사이트"로 보이길 원하는가?

이번 office hours의 결론은 후자다. 구현 단계에서는 이 결론을 기준으로 홈 우선순위와 문구를 바꾼다.
