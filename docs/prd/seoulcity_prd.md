# 서울시티 PRD

## 1. 문서 목적

이 문서는 서울시티의 제품 방향과 요구사항을 정리한 기준 문서입니다. 기능 구현 방법보다, 이 서비스가 누구의 어떤 문제를 해결해야 하는지와 제품 판단 기준을 고정하는 데 목적이 있습니다.

요구사항이 추가되거나 제품 방향이 바뀌면 이 문서를 함께 갱신합니다.

## 2. 제품 개요

서울시티는 서울 생활 정보를 한곳에서 찾기 위한 로컬 정보 서비스입니다. 사용자는 행사, 공공 혜택, 먹거리, 블로그 정보글, 구별·상황별 가이드를 함께 보면서 서울에서 갈 곳이나 받을 수 있는 정보를 빠르게 고를 수 있어야 합니다.

현재 제품은 Next.js App Router 기반 정적 사이트로 운영되며, 공공 데이터 JSON과 Markdown 블로그 글을 함께 사용합니다.

## 3. 문제 정의

서울 생활 정보는 여러 공식 사이트와 데이터 포털에 흩어져 있습니다. 사용자는 주말 나들이, 비 오는 날 실내 코스, 아이와 갈 곳, 무료 행사, 공공 혜택처럼 실제 상황 중심으로 정보를 찾지만, 원천 데이터는 행정 기준으로 나뉘어 있어 바로 선택하기 어렵습니다.

해결해야 할 핵심 문제는 다음과 같습니다.

- 행사, 혜택, 먹거리, 정보글이 서로 분리되어 있어 한 번에 비교하기 어렵습니다.
- 공식 데이터는 신뢰할 수 있지만, 일반 사용자가 탐색하기에는 딱딱하고 맥락이 부족합니다.
- 사용자는 “어디를 갈까”, “이번 주에 뭐가 있지”, “무료로 볼 만한 게 있나”처럼 상황 중심으로 묻습니다.
- 공공정보는 최신성, 출처, 공식 확인 링크가 명확하지 않으면 신뢰하기 어렵습니다.

## 4. 목표

서울시티의 목표는 사용자가 검색을 여러 번 반복하지 않고도 서울 생활 정보를 선택할 수 있게 돕는 것입니다.

제품 목표:

- 서울 행사, 혜택, 먹거리, 블로그 정보글을 하나의 탐색 경험으로 묶습니다.
- 상황별, 구별, 검색 기반 탐색을 모두 지원합니다.
- 공공 데이터의 출처와 수집 기준을 사용자가 확인할 수 있게 합니다.
- 정적 사이트 구조를 유지해 빠르고 안정적으로 배포합니다.

## 5. 대상 사용자

주요 사용자:

- 서울에서 주말이나 퇴근 후 갈 곳을 찾는 사람
- 아이와 갈 만한 장소를 찾는 보호자
- 비 오는 날 실내 코스나 무료·저비용 코스를 찾는 사람
- 서울시 또는 자치구 공공 혜택을 놓치고 싶지 않은 사람
- 특정 구를 기준으로 행사, 혜택, 먹거리를 같이 보고 싶은 사람

보조 사용자:

- 사이트 운영자
- 콘텐츠 생성 및 데이터 갱신을 관리하는 개발자
- 잘못된 정보나 오래된 정보를 제보하려는 사용자

## 6. 핵심 사용자 시나리오

### 6.1 오늘 서울에서 볼 만한 것을 빠르게 찾기

사용자는 홈에서 오늘 진행 중인 행사, 무료 행사, 실내 추천, 이번 주 마감 혜택을 바로 확인할 수 있어야 합니다.

### 6.2 상황별로 코스 찾기

사용자는 “아이와”, “비 오는 날”, “무료·저비용” 같은 상황을 기준으로 관련 가이드와 글을 볼 수 있어야 합니다.

### 6.3 구별로 탐색하기

사용자는 특정 구의 주요 권역, 최신 행사, 연결 혜택, 먹거리 후보를 한 페이지에서 볼 수 있어야 합니다.

### 6.4 통합 검색하기

사용자는 하나의 검색어로 행사, 혜택, 먹거리, 블로그 정보글을 함께 찾을 수 있어야 합니다.

### 6.5 공식 정보 확인하기

사용자는 행사나 혜택 상세에서 데이터 출처, 수집 시각, 공식 링크를 확인할 수 있어야 합니다.

## 7. 현재 제품 범위

현재 포함 범위:

- 서울시 행사·축제 목록과 상세 페이지
- 공공데이터포털 기반 지원금·혜택 목록과 상세 페이지
- 서울 구별 먹거리 가이드와 카카오맵 검색 연결
- 구별 가이드와 구별 상세 페이지
- 상황별 가이드 페이지
- Markdown 기반 서울 생활 정보 블로그
- 행사, 혜택, 먹거리, 블로그 통합 검색
- 오늘의 서울 요약 지표
- 서비스 소개, 운영 고지, 개인정보처리방침, 문의 페이지
- Cloudflare AI 기반 간단한 서울 정보 도우미 챗봇

현재 제외 범위:

- 예약, 결제, 신청 대행
- 사용자 계정과 개인화 추천
- 실시간 위치 기반 추천
- 공공기관 원문을 대체하는 법적·행정적 안내
- 모든 서울 25개 구에 대한 동일 수준의 에디토리얼 가이드 보장
- 실시간 데이터 조회형 애플리케이션

## 8. 기능 요구사항

### 8.1 홈

- 사용자는 홈에서 서비스의 목적을 바로 이해할 수 있어야 합니다.
- 사용자는 통합 검색 입력창을 홈 상단에서 사용할 수 있어야 합니다.
- 사용자는 상황별 코스, 추천 글, 오늘의 서울 요약, 행사·혜택 하이라이트를 볼 수 있어야 합니다.

### 8.2 행사

- 사용자는 서울 행사·축제 목록을 볼 수 있어야 합니다.
- 사용자는 진행 중, 무료, 실내 추천 같은 프리셋으로 행사 목록을 좁힐 수 있어야 합니다.
- 사용자는 구별 필터로 행사를 탐색할 수 있어야 합니다.
- 사용자는 행사 상세에서 일정, 장소, 요금, 문의, 출처, 공식 링크를 확인할 수 있어야 합니다.

### 8.3 혜택

- 사용자는 서울시와 자치구, 서울 공공기관 관련 혜택을 볼 수 있어야 합니다.
- 사용자는 분야와 지역으로 혜택을 필터링할 수 있어야 합니다.
- 사용자는 이번 주 마감 혜택을 빠르게 볼 수 있어야 합니다.
- 사용자는 혜택 상세에서 지원 대상, 신청 기한, 신청 방법, 지원 내용, 문의처, 공식 링크를 확인할 수 있어야 합니다.

### 8.4 먹거리

- 사용자는 서울 구별 먹거리 개요를 볼 수 있어야 합니다.
- 사용자는 상세 가이드가 있는 구는 내부 페이지로 이동할 수 있어야 합니다.
- 상세 가이드가 없는 구는 카카오맵 검색으로 연결할 수 있어야 합니다.
- 음식점 후보는 추천 순위가 아니라 공공데이터 기반 참고 후보임을 명확히 알려야 합니다.

### 8.5 구별 가이드

- 사용자는 구별 주요 권역과 반나절 코스 예시를 볼 수 있어야 합니다.
- 사용자는 해당 구와 연결되는 행사, 혜택, 블로그 글, 먹거리 정보를 함께 볼 수 있어야 합니다.
- 사용자는 구별 공식 확인 링크를 볼 수 있어야 합니다.

### 8.6 상황별 가이드

- 사용자는 아이와, 비 오는 날, 무료·저비용 같은 상황별 페이지를 볼 수 있어야 합니다.
- 각 상황별 페이지는 판단 기준, 체크리스트, 관련 글, 통합 검색 링크를 포함해야 합니다.

### 8.7 블로그

- 블로그 글은 Markdown으로 관리합니다.
- 공개 글은 `region: 서울`, `sourceType: 정보글` 기준을 만족해야 합니다.
- 블로그 상세에는 작성일, 최종 업데이트, 카테고리, 태그, AI 활용 안내, 공식 확인 안내가 있어야 합니다.

### 8.8 통합 검색

- 사용자는 행사, 혜택, 먹거리, 블로그를 하나의 검색창에서 찾을 수 있어야 합니다.
- 검색 결과는 유형, 지역, 현재 참고 가능 여부로 필터링할 수 있어야 합니다.
- 먹거리 결과처럼 외부로 이동하는 항목은 외부 링크임을 안전하게 처리해야 합니다.

### 8.9 운영 고지

- 사용자는 서비스 소개, 개인정보처리방침, 서비스 안내, 광고·분석 고지, 문의 및 정정 요청 페이지에 접근할 수 있어야 합니다.
- 정정 요청 경로는 개인정보가 섞일 수 있는 요청과 공개 오류 제보를 구분해야 합니다.

## 9. 데이터 요구사항

주요 데이터 소스:

- 서울 열린데이터광장 `서울시 문화행사 정보`
- 공공데이터포털 `대한민국 공공서비스(혜택) 정보`
- 서울 열린데이터광장 `서울시 일반음식점 인허가 정보`
- `src/content/posts`의 Markdown 정보글

데이터 저장 기준:

- 행사: `public/data/events/index.json`과 `public/data/events/items/*.json`
- 혜택: `public/data/benefits/index.json`과 `public/data/benefits/items/*.json`
- 먹거리: `public/data/food/index.json`과 `public/data/food/items/*.json`
- 통합 검색: `public/data/search/index.json`
- 챗봇용 검색: `public/data/search-index.json`

데이터 품질 기준:

- 수집 시각과 출처를 보존합니다.
- 상세 페이지에는 원본 식별자나 공식 링크를 가능한 한 노출합니다.
- 만료된 행사는 목록에서 숨깁니다.
- 혜택은 신청기한과 상시신청 여부를 구분합니다.
- 먹거리 데이터는 방문 추천이 아니라 공공데이터 기반 후보로 설명합니다.

## 10. 콘텐츠 요구사항

- UI 문구는 한국어를 기본으로 유지합니다.
- 공공정보는 과장하지 않고 참고용 안내로 표현합니다.
- AI 보조 작성 글은 AI 활용 사실과 공식 확인 필요성을 알려야 합니다.
- 블로그 글은 서울 생활 맥락과 실제 선택 기준을 제공해야 합니다.
- 공식 원문에 없는 내용을 확정적으로 쓰지 않습니다.

## 11. SEO와 신뢰 요구사항

- 주요 페이지는 고유한 title과 description을 가져야 합니다.
- 사이트맵은 검색에 노출할 canonical URL만 포함해야 합니다.
- `robots.txt`는 sitemap 위치를 제공해야 합니다.
- 블로그 상세, 홈, 주요 가이드 페이지에는 화면에 실제로 보이는 정보와 일치하는 JSON-LD만 사용합니다.
- 검색 결과 URL과 필터 쿼리 URL은 별도 색인 정책을 정해야 합니다.
- 사용자가 공식 링크와 정정 요청 경로를 찾기 쉬워야 합니다.

## 12. 비기능 요구사항

- 정적 배포 구조를 유지합니다.
- 빌드 전 검색 인덱스를 생성해야 합니다.
- 외부 API 키가 없어도 일반 개발과 정적 페이지 확인은 가능해야 합니다.
- 데이터 수집 스크립트는 API 키와 네트워크 실패를 운영자가 진단할 수 있게 해야 합니다.
- 민감한 값은 코드와 문서에 노출하지 않습니다.

## 13. 성공 지표

초기 제품 성공 기준:

- 사용자가 홈에서 주요 탐색 경로를 이해할 수 있습니다.
- 사용자가 통합 검색으로 행사, 혜택, 먹거리, 글을 함께 찾을 수 있습니다.
- 사용자가 상세 페이지에서 공식 확인 링크나 출처를 찾을 수 있습니다.
- 운영자가 데이터 수집 시각과 자동화 흐름을 파악할 수 있습니다.
- 새 요구사항이 들어왔을 때 PRD, README, 운영 문서 중 어떤 문서를 갱신해야 하는지 판단할 수 있습니다.

정량 지표 후보:

- 통합 검색 사용률
- 행사·혜택 상세 페이지 클릭률
- 공식 링크 클릭률
- 블로그 글 유입과 체류 시간
- 데이터 수집 실패 빈도
- 정정 요청 건수와 처리 시간

## 14. 문서 운영 규칙

- 제품 범위나 타깃 사용자가 바뀌면 이 PRD를 갱신합니다.
- 데이터 수집 방식이 바뀌면 데이터 파이프라인 문서를 함께 갱신합니다.
- 실행·배포 방식이 바뀌면 README를 함께 갱신합니다.
- 날짜별 작업 이력은 `history/`에 기록합니다.
- 사용자에게 보이는 주요 변경은 `change_log.md`에 정리합니다.

## 15. 열린 질문

- 서울 25개 구 전체에 대해 내부 구별 가이드를 제공할 것인가?
- 검색 결과 URL을 색인하지 않을 것인지, 일부 랜딩 검색만 색인할 것인지 결정해야 합니다.
- 챗봇은 블로그 기반 안내만 유지할 것인지, 행사·혜택·먹거리 검색까지 확장할 것인지 결정해야 합니다.
- 블로그는 보조 콘텐츠로 둘 것인지, SEO 유입의 핵심 채널로 키울 것인지 결정해야 합니다.
- 데이터 최신성 경고를 어떤 기준으로 사용자에게 노출할지 결정해야 합니다.
